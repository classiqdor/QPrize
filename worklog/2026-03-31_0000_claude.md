# Session 2026-03-31 00:00 — fast_mul exploration and abandonment

## Context

Continuing from the 2026-03-30 session where attempt_012 (sq_lookup, 105,554 CX) became the best genuine ECDLP implementation. The goal was to further reduce CX by replacing `modular_multiply` (2527 CX) with a cheaper schoolbook decomposition (`fast_mul`).

## Attempts this session

### Attempts 015–018: Python/Classiq API issues in fast_mul

The `fast_mul` idea: decompose `a*b mod p` bit-by-bit using 1D Const[QNum] lookups + controlled `modular_add_inplace`. All four attempts failed before synthesis:

- **015**: `SyntaxError` — `temp ^= subscript(...)` is an augmented assignment statement, cannot appear inside a lambda/tuple expression. `^=` is overridden by Classiq's QNum to handle lookup results; it's not equivalent to `inplace_xor`.
- **016**: `ClassiqInternalExpansionError: CParamScalar has no get_handle_binding` — tried `inplace_xor(temp, subscript(...))` as a workaround, but `subscript(tbl, b)` returns a `CParamScalar`, not a quantum handle. Only `QNum.__ixor__` handles CParamScalar lookup results.
- **017**: `ClassiqExpansionError: UnboundLocalError` — switched to `def compute_i(ii=i): ... temp ^= ...` per iteration. But `temp ^= ...` inside a def makes Python treat `temp` as a local variable (augmented assignment = local binding), causing UnboundLocalError when it tries to read the outer `temp` before assignment.
- **018**: `ClassiqExpansionError: QNum not subscriptable` — switched to explicit double-XOR (allocate temp once, XOR-load, controlled add, XOR-unload). Needed `a[i]` for bit-level control, but `a: QNum` doesn't support indexing. Fix: use `a: QArray[QBit]`.

### Attempt 019: fast_mul synthesized but WORSE

With `a: QArray[QBit]`, attempt_019 synthesized successfully:
- **Qubits**: 32 (vs 28 for attempt_012 — 4 extra from `temp` register in fast_mul)
- **Depth**: 167,196 (vs 105,378)
- **CX**: 146,402 (vs 105,554 — **38% worse**)
- **Execution**: failed — 32 qubits > 28-qubit simulator limit
- **Synthesis time**: 840s (vs 506s for attempt_012)

The root cause: `control(a[i], modular_add_inplace(p, temp, result))` is a CONTROLLED modular addition, which costs ~1000-1500 CX each. With 4 bits × 4 iterations per fast_mul call, the total per fast_mul is ≥ modular_multiply (2527 CX). Plus extra qubit overhead.

## Key lessons

1. **Classiq's modular_multiply (2527 CX) is the best available multiplier** — all attempts to replace it with user-level decompositions have been worse:
   - mul_lookup 2D in COMPUTE (011): 136k CX
   - slope_lookup 2D outside COMPUTE (014): 128k CX
   - fast_mul schoolbook (019): 146k CX

2. **The 1D vs 2D rule is confirmed**: Only 1D lookups with Const[QNum] (single quantum arg, not modified) are efficient. sq_lookup (step 5) is the only win because `slope^2 mod p` depends on a single register.

3. **Fundamental limit**: All quantum-quantum multiplications (both operands in superposition) require 2D information and cannot be reduced to 1D lookups.

4. **Python gotcha with `^=` in closures**: `QNum.__ixor__` handles lookup results but can't be used in nested function scopes where Python would make the variable local.
